# External Agent Benchmark Findings

This document captures the current benchmark findings for the external agent adapter lane.

It is meant to answer three questions quickly:

1. What exactly am I testing?
2. How is each tool being run?
3. What did the latest results show?

For the adapter design and contract itself, see [External Agent Benchmark Adapters](./external-agent-benchmark-adapters.md).

## What I am testing

I am benchmarking agent CLIs as black-box coding tools on the same controlled fixture task.

For each run, I:

1. create a fresh workspace from a benchmark fixture
2. give the agent the same task prompt
3. let the agent operate normally in that workspace
4. run the same validator against the finished workspace
5. record runtime, cost, usage, and changed files when available

This is not a mocked comparison. Each tool is doing real work in a real temporary workspace.

## Current scenario

The current comparison scenario is `external-static-site`.

The task is:

- read `requirements.md`
- inspect the existing workspace
- create a dependency-free `index.html`
- make sure `npm run build` and `npm run test` pass
- summarize the files changed

The validator checks that the resulting page contains:

- `<title>Hello Aura</title>`
- `Hello from Aura`
- `data-testid="tagline"`
- `This page was generated by Aura.`

This gives me a small but real coding task that still exercises:

- file reading
- workspace inspection
- code generation
- verification
- final reporting

## How each tool is run

### Aura

Aura is run through the real local Aura harness over websocket.

I do not mock the loop. The adapter opens a real harness session and sends a single benchmark turn that instructs the agent to:

- read `requirements.md`
- inspect the workspace
- implement the page exactly as specified
- keep the project dependency-free
- verify the output before finishing

This keeps Aura on its native runtime path and preserves rich harness telemetry.

### Claude Code

Claude Code is run as a CLI in the prepared workspace with:

- non-interactive `--print`
- `stream-json`
- validator-based grading

The benchmark runner now parses Claude's transcript output so I can surface:

- runtime
- direct reported USD cost
- input/output tokens
- cache read/write tokens
- provider/model

One environment caveat matters here:

- if `CLAUDE_CODE_USE_VERTEX=1` is present but `gcloud` is not authenticated, Claude will fail
- the stable local path for this benchmark is to preserve `ANTHROPIC_API_KEY` and unset the Vertex env vars

### Codex

Codex is run as a CLI in the prepared workspace with:

- `codex exec`
- JSONL output
- validator-based grading

The benchmark runner now parses Codex's transcript output so I can surface:

- runtime
- input/output tokens
- cached input tokens
- provider/model

Codex cost can also be surfaced, but only if the benchmark run specifies the model explicitly. That matters because Codex's JSONL output exposes usage, but not always the model name.

## Why these numbers are useful

This lane answers a different question from the direct Aura harness benchmarks.

The direct Aura harness benchmarks tell me:

- whether Aura cache/context logic is improving
- whether Aura is getting cheaper or faster with the same runtime

The external adapter lane tells me:

- how Aura compares to other serious coding-agent CLIs on the same task

So this benchmark is mainly about:

- outcome quality
- runtime
- cost
- observability quality

## Current read of the results

On this scenario:

- Aura was the fastest
- Aura was the cheapest
- Claude Code was second and fairly close
- Codex passed, but was materially slower and more expensive on this task

The strongest honest conclusion is:

- Aura is currently competitive even against Claude Code on the same Anthropic model family
- Codex is currently behind on this specific benchmark

## Important caveats

These numbers are useful, but they are not the final word.

The current caveats are:

- this is still a small single-shot task
- Aura and Claude are on `claude-opus-4-6`, while Codex is on `gpt-5.4`
- external CLI telemetry is best-effort and weaker than Aura's own harness telemetry
- this benchmark should be treated as one scenario, not a universal ranking

So I should describe this as:

- a strong early black-box benchmark
- not yet a full multi-scenario vendor ranking

## What I fixed to make this benchmark more honest

The first version of the external adapter lane left some cost fields blank.

I fixed that by:

- parsing Claude Code `stream-json` transcripts for direct cost and token usage
- parsing Codex JSONL transcripts for token usage
- pricing Codex runs when the model is provided explicitly
- keeping unsupported fields as `null` instead of inventing fake parity

That means the current results are materially more trustworthy than the first adapter prototype.
